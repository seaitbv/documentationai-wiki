## Overview

This gap analysis compares PRD v1.3.1 (04-03-2026) against the current ARMS implementation using a 6-perspective methodology:

1. **Functional** — feature completeness against PRD requirements
2. **Technical** — code-level implementation gaps
3. **Solution Architecture** — structural and integration gaps
4. **Database** — schema, constraints, and seed data gaps
5. **Next.js** — frontend-specific gaps (UI/UX, navigation, i18n)
6. **Devil's Advocate** — contradictions, ambiguities, and stakeholder decisions

  <Expandable title="4. Business logic gaps" default-open="false">
    ### KM invoicing — not implemented

    <Callout kind="danger">
      **Severity: Critical** — entire billing unit missing. The `generateInvoiceProposals` function only accepts `unitType: "day" | "month"`. KM is entirely excluded.
    </Callout>

    Missing functionality:
    - KM proposal generation (determine start/end km from `km_registration` table)
    - KM billing calculation (`km_driven x unit_price_rental`)
    - Red marking when km registration data is missing near billing period boundaries (FR-INVP-003)
    - `invoice_km_lock` creation during KM invoicing (table exists but never populated)
    - Exclusion of non-driving days for KM contracts (FR-INVP-005)
    - Exclusion of KM allowance for km-unit contracts (Section 7.4)

    ### KM yearly allowance — not implemented

    <Callout kind="alert">
      **Severity: High** — revenue-relevant calculation missing. Parameters exist (`KM_YEARLY_MAX = 100000`, `KM_SURPLUS_PRICE_EUR = 0.30`) but no code consumes them.
    </Callout>

    Missing functionality:
    - Annual KM usage computation per contract/trailer
    - Comparison against 100,000 km threshold
    - Surcharge calculation at 0.30 EUR/km for overage
    - Surcharge invoice line generation

    ### Forfait logic

    <Callout kind="success">
      **Implemented.** Forfait correctly uses total contract duration (not invoice period). Parameters `FORFAIT_1_DAY_EUR` (120) and `FORFAIT_2_DAYS_EUR` (170) are configurable.
    </Callout>

    ### Deposit/advance gaps (Section 7.6)

    | Gap | PRD Requirement | Current Implementation |
    |-----|----------------|----------------------|
    | Combined invoice | Deposit + advance on 1 invoice with 2 lines (FR-DEP-001) | Creates 2 separate invoices |
    | Due date | `contract_start - N days` (default 14) (FR-DEP-002) | Uses `today + 30 days` |
    | Negative line convention | qty less than 0, price greater than 0 (Peppol compatible) (FR-DEP-003) | Uses `qty: 1, price: -amount` |
    | Saldo tracking | Only real invoices affect saldo (FR-DEP-004) | No distinction between proposals and invoices |
    | Default advance amount | day=30xprice, month=1xprice, km=6000kmxprice | Default 0, no auto-calculation |

  <Expandable title="8. Security/RBAC gaps" default-open="false">
    ### Role model

    PRD defines 5 roles: Admin, Verhuurmedewerker (commercial), Vlootbeheerder (fleet_manager), Facturatie (accounting), Management (read_only).

    Current implementation uses: `admin`, `commercial`, `accounting`, `fleet_manager`, `read_only`.

    <Callout kind="info">
      Role names map but "Management" = `read_only` is a semantic mismatch. Management may need KPI/reporting access beyond pure read-only. Requires stakeholder clarification.
    </Callout>

    ### Page-level RBAC

    | Gap | Details |
    |-----|---------|
    | Binary admin check | Most pages only check `isAdmin` for write permissions. Full 5-role matrix not enforced at component level. |
    | Missing module-level write guards | `commercial` should write offers/contracts but not fleet. `fleet_manager` should write fleet but not invoices. Currently: admin = full write, non-admin = limited write. |
    | No per-company data segregation in UI | RLS policies exist at DB level, but the UI does not filter by user's company. All users see all companies' data. |

  <Expandable title="10. Contradictions and ambiguities" default-open="false">
    <Callout kind="alert">
      These findings require **stakeholder decisions** before implementation can proceed.
    </Callout>

    ### 10.1 Customer-contact relationship: 1:N vs M:N

    **PRD**: Bijlage A lists a `customer_contact` linking table (implying M:N).
    **Current**: `contact.customer_id` FK makes it 1:N.
    **Decision needed**: Can one contact serve multiple customers? If not, current 1:N is correct.

    ### 10.2 Invoice-contract nullability

    **PRD** (FR-INV-MAN-001): "facturen moeten altijd gelinkt zijn aan een contract"
    **Current**: `invoice.contract_id` is nullable (migration 000014 made it optional).
    **Decision needed**: Should standalone invoices be allowed?

    ### 10.3 Deposit/advance: 1 invoice vs 2 invoices

    **PRD** (FR-DEP-001): "Indien beide gevraagd: 1 factuur met 2 factuurlijnen"
    **Current**: Creates 2 separate invoices.
    **Decision needed**: Confirm PRD is correct (1 invoice with 2 lines). Affects invoice numbering and Exact Online mapping.

    ### 10.4 Negative line convention

    **PRD** (FR-DEP-003): qty less than 0, price greater than 0 (Peppol compatible)
    **Current**: `qty: 1, price: -amount`
    **Decision needed**: Which convention does Exact Online / Peppol require?

    ### 10.5 Verhuurstatus: computed vs manual

    **PRD** (FR-FLEET-001/003): Verhuurstatus is time-dependent, derived from planning events.
    **Current**: Manually-set dropdown value.
    **Decision needed**: Auto-compute from active contracts, or keep manual with a separate computed badge?

    ### 10.6 Contract status auto-transitions

    **PRD** (FR-CON-005) implies multiple auto-transitions:
    - Geaccepteerd to In verhuur (when start date is before or equal to today) — **implemented**
    - In verhuur to Te controleren (after real end date) — **not implemented**
    - Te controleren to Schadeverwerking (if damage) — **not implemented**, state doesn't exist

    **Decision needed**: Should all be auto-transitions or manual?

    ### 10.7 Factuurnummering: internal vs Exact

    **PRD** (FR-INVNUM-001): "Tool bewaart intern factuurnummer en Exact-factuurnummer"
    **Current**: `invoice_number` (internal) + `exactonline_ref` (self-generated).
    **Decision needed**: How does Exact Online assign its own number? Does the tool need to store it?

    ### 10.8 PDF generation technology

    **PRD** (FR-COMM-001, FR-CON-006): Offerte/contract PDF generation from templates.
    **Current**: Template upload/storage works, no PDF engine.
    **Decision needed**: Which library? puppeteer/chromium (HTML to PDF), pdf-lib, react-pdf, or server-side? Affects template format.

    ### 10.9 Notification triggers: real-time vs batch

    **PRD** (FR-NOTIF-001, FR-SET-001): Parameters trigger notifications.
    **Current**: `notification_log` table exists, logic for checking conditions present, execution model unclear.
    **Decision needed**: Cron job (Supabase Edge Function), database triggers, or application-level event handlers?

    ### 10.10 Outlook: desktop vs web

    **PRD** (Section 12): "Technische aanpak voor Outlook draft (desktop vs web)"
    **Current**: Microsoft Graph API (web-based cloud mailbox drafts).
    **Decision needed**: Is Graph API sufficient, or do users expect desktop Outlook integration?

    ### 10.11 STAS as invoice-to

    **PRD** (FR-CON-004): "STAS kan ook gekozen worden (bv. garantie)"
    **Current**: `invoice_to_id` FK points to `customer` table.
    **Decision needed**: Is STAS a customer record, or does it need special handling?

    ### 10.12 Repair order status sync

    **PRD** (TR-REPAIR-001): "Synchronisatie van repair order status"
    **Current**: Only `repair_order_ref` text field.
    **Decision needed**: Which external system? What statuses? Polling or manual entry?
