## Overview

ARMS supports three invoice calculation modes based on the contract's billing unit: **day**, **month**, and **km**. Each mode generates specific invoice lines with distinct pricing logic. An anti-double-invoicing system prevents the same period or km range from being billed twice.

## Calculation modes

| Mode | Unit | Use case |
|------|------|----------|
| Day | `dag` | Short-term and standard rentals billed per day |
| Month | `maand` | Long-term rentals billed monthly with pro-rata |
| Km | `km` | Distance-based rentals billed per kilometer driven |

## Common concepts

All modes share the concept of **effective dates**, which determine the actual rental period:

```
effective_start = COALESCE(rental_start_real, rental_start_estimated)
effective_end   = COALESCE(rental_end_real,   rental_end_estimated)
```

The billing period is then clamped to the contract's effective period:

```
billable_from = MAX(billing_period_start, effective_start)
billable_to   = MIN(billing_period_end,   effective_end)

IF billable_from > billable_to THEN skip this contract
```

## Day-based calculation

### Billable days

```
overlap_days = (billable_to - billable_from) + 1
non_driving_days = SUM of Non_Driving_Days overlapping [billable_from, billable_to]
billable_days = overlap_days - non_driving_days
```

### Forfait logic

Forfait pricing applies when the **total contract duration** (not the billing period) is 1 or 2 days:

```
total_contract_days = (effective_end - effective_start) + 1

IF total_contract_days == 1:
    Single line: Forfait 1 dag = 120 EUR (parameter FORFAIT_1_DAY_EUR)
    Unit = "stuk"

ELIF total_contract_days == 2:
    Single line: Forfait 2 dagen = 170 EUR (parameter FORFAIT_2_DAYS_EUR)
    Unit = "stuk"

ELSE:
    Standard day-rate calculation (see below)
```

<Callout kind="tip">
  Forfait amounts are configurable via the [parameter system](/technical/business-logic/parameterization). The default values are 120 EUR (1 day) and 170 EUR (2 days).
</Callout>

### Standard day-rate invoice lines (3+ days)

| Line | Description | Quantity | Unit price | VAT | Discount |
|------|-------------|----------|------------|-----|----------|
| 1 | Huur (Rental) | `overlap_days` | `unit_price_rental` | Contract VAT % | Yes |
| 2 | Aftrek niet-rijdagen (NDD deduction) | `-non_driving_days` | `unit_price_rental` | Contract VAT % | Yes |
| 3 | Verzekering (Insurance) | `overlap_days` | `unit_price_insurance` | 0% | No |
| 4 | Aftrek verzekering niet-rijdagen | `-non_driving_days` | `unit_price_insurance` | 0% | No |

Lines 2 and 4 only appear when `non_driving_days > 0`. Lines 3 and 4 only appear when `unit_price_insurance > 0`.

<Callout kind="info">
  When forfait pricing applies (1 or 2 day contracts), lines 1-4 are replaced by a single forfait line. Insurance lines are still added separately.
</Callout>

### Calculation pseudocode

```typescript
// For contracts with 3+ total days
rental_multiplier = 1 - (discount_pct / 100)

rental_line_total = overlap_days * unit_price_rental * rental_multiplier
ndd_deduction     = -non_driving_days * unit_price_rental * rental_multiplier
insurance_total   = overlap_days * unit_price_insurance
insurance_ndd     = -non_driving_days * unit_price_insurance

total_excl_vat = rental_line_total + ndd_deduction + insurance_total + insurance_ndd
```

## Month-based calculation

Month billing uses a pro-rata fraction of the month, rounded up to 2 decimal places.

### Pro-rata calculation

```
days_used     = (billable_to - billable_from) + 1
days_in_month = total days in the selected calendar month

months_fraction = CEILING(days_used / days_in_month, 2 decimals)
```

**Example:** Contract starts 26 February 2026. February has 28 days.
- Days used: 26, 27, 28 = 3 days
- Pro-rata: 3 / 28 = 0.1071... rounded up to **0.11 month**

### Month invoice lines

| Line | Description | Quantity | Unit price | VAT | Discount |
|------|-------------|----------|------------|-----|----------|
| 1 | Huur (Rental) | `months_fraction` | `unit_price_rental` | Contract VAT % | Yes |
| 2 | Verzekering (Insurance) | `months_fraction` | `unit_price_insurance` | 0% | No |

<Callout kind="info">
  Non-driving days are **not applicable** for month-based billing. Only day-based billing deducts non-driving days.
</Callout>

### Calculation pseudocode

```typescript
rental_multiplier = 1 - (discount_pct / 100)

rental_amount    = months_fraction * unit_price_rental * rental_multiplier
insurance_amount = months_fraction * unit_price_insurance

total_excl_vat = rental_amount + insurance_amount
```

## Km-based calculation

Km billing uses odometer readings from the `Km_Registration` table.

### Km lookup

```
km_start = most recent Km_Registration for plate_number with date <= period_start
km_end   = most recent Km_Registration for plate_number with date <= period_end

driven_km = km_end.km_amount - km_start.km_amount
```

<Callout kind="alert">
  If the last km registration date is before `period_end` or before `effective_end`, the proposal row is flagged in red with "Missing km data". The user should register km readings before generating the invoice.
</Callout>

### Km invoice lines

| Line | Description | Quantity | Unit price | VAT | Discount |
|------|-------------|----------|------------|-----|----------|
| 1 | Huur (Rental) | `driven_km` | `unit_price_rental` | Contract VAT % | Yes |
| 2 | Verzekering (Insurance) | `driven_km` | `unit_price_insurance` | 0% | No |

## Advance offset

When a contract has a prepaid advance (`advance_amount`), it is gradually offset against recurring invoices:

```
remaining = advance_amount - advance_offset_amount

IF remaining > 0 AND invoice_subtotal > 0:
    offset_applied = MIN(remaining, invoice_subtotal)
    Add line: "Voorschot verrekening" with amount = -offset_applied (0% VAT)
    Update contract: advance_offset_amount += offset_applied
```

This ensures the advance is deducted from the first recurring invoice(s) until fully consumed.

## Anti-double-invoicing

ARMS prevents duplicate billing through period locks and km locks.

### Invoice_Period_Lock

For day and month billing, each invoice creation writes a lock record:

| Column | Description |
|--------|-------------|
| `contract_id` | The contract being billed |
| `invoice_id` | The generated invoice |
| `locked_from` | Start of the billed period |
| `locked_to` | End of the billed period |

When generating proposals, the system checks existing locks and classifies each contract:

| Status | Meaning |
|--------|---------|
| `normal` | No locks overlap with the billing period |
| `partially_invoiced` | Some of the period is already locked |
| `fully_invoiced` | The entire period is covered by locks |

Fully invoiced contracts are excluded from new invoice creation. A race-condition guard re-checks locks immediately before inserting the invoice.

### Invoice_Km_Lock

For km billing, a similar mechanism tracks which km ranges have been invoiced, preventing the same kilometers from being billed twice.

## Invoice grouping

When creating invoices from proposals, the system groups proposals by **customer + company**. All proposals for the same customer and company are combined into a single invoice with multiple line items.
